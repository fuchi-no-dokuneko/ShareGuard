Document title:
Canonical Share — Android Sequential Privacy-Normalization Pipeline
Detailed Product, UX, Block, Verification and Agent-Implementation Specification

Document version:
Draft v1.0

Document date:
2026-07-15

Primary audience:
- Product owner
- Android architects
- Coding agents
- Security reviewers
- QA and adversarial-testing engineers
- UX designers

Primary implementation language:
JAVA

Target UI framework:
Jetpack Compose

Core product rule:
The application accepts exactly one primary input item per session:
- text, or
- image.

The execution model is a typed, ordered, sequence-only pipeline.
There is no free graph, no branch connector, no loop connector and no arbitrary node-to-node wiring in the user interface.

The pipeline may build an Output Bundle containing:
- canonical text;
- a rebuilt image;
- a verification report;
- share-ready Android objects.

This does not violate the sequence-only rule because each block consumes and returns one Execution Context.
Blocks append or replace typed artifacts inside that context in a deterministic order.

Terminology:
- Source: the untrusted text or image received by the app.
- Finding: a detected condition that may carry identity or provenance information.
- Change: a transformation actually applied by the app.
- Canonical Document: a semantic intermediate representation that does not inherit source bytes or source pixels by default.
- Rebuilt Image: a newly rendered image generated from the Canonical Document.
- Appearance-Preserving Image: a transformed derivative that still depends on source pixels.
- Assurance: a constrained claim about what processing occurred; it is not proof that every unknown watermark was removed.
- Review Gate: a mandatory user decision point when automatic processing could alter meaning.
- Verification Gate: a post-output check that reopens and inspects the final artifact.


====================================================================================================
1. EXECUTIVE PRODUCT DECISION
====================================================================================================
Decision:
Use a ComfyUI-inspired block vocabulary and execution visibility, but do not copy the unrestricted node-canvas interaction.

Approved interaction model:
- vertical ordered block list;
- one source;
- one execution cursor;
- typed block compatibility;
- mandatory security blocks;
- optional policy blocks;
- no cycles;
- no user-created ports;
- no parallel branches;
- no hidden execution path;
- a visible final assurance result.

Why sequence-only is appropriate:
1. Security order is predictable.
2. Every output has a reproducible processing history.
3. Coding agents can implement one block at a time.
4. Test cases can assert the exact block order.
5. Mandatory controls cannot be bypassed by accidental graph rewiring.
6. Mobile UX remains understandable.
7. Workflow migrations are simpler.
8. Support reports can identify the exact block and version that created a change.
9. A failed block has a single predecessor and successor.
10. The app can explain, in plain language, what happened to the content.

The product should not be marketed as:
- guaranteed watermark removal;
- guaranteed anonymity;
- complete anti-forensics;
- protection against platform account, network, timing or behavioural correlation;
- proof that the source cannot be identified.

Recommended positioning:
“Convert shared text or screenshots into a newly normalized representation, reducing inherited formatting, metadata, Unicode, URL, font and pixel-level provenance signals.”

Primary high-assurance outputs:
- Canonical Text
- Rebuilt Image

Experimental lower-assurance output:
- Appearance-Preserving Image


====================================================================================================
2. PRODUCT GOALS, NON-GOALS AND TRUST BOUNDARY
====================================================================================================
2.1 Goals

G-001  Accept text from Android sharing, paste and direct entry.
G-002  Accept one image from Android sharing or the system Photo Picker.
G-003  Show the detected input type prominently.
G-004  Let the user choose Text, Rebuilt Image, Both, or an explicitly lower-assurance appearance-preserving output.
G-005  Expose processing as an ordered set of inspectable blocks.
G-006  Normalize Unicode, whitespace, punctuation, script-confusables and URLs.
G-007  Extract image text and layout without treating OCR as automatically correct.
G-008  Rebuild images on a fresh canvas with bundled assets and controlled serialization.
G-009  Avoid carrying source metadata into outputs.
G-010  Reopen final outputs and verify their actual contents.
G-011  Keep processing local by default.
G-012  Avoid stable installation-specific random signatures.
G-013  Delete transient source and intermediate artifacts.
G-014  Produce an understandable change report.
G-015  Make every security claim traceable to executed blocks and verification results.

2.2 Non-goals

NG-001  Preventing a destination app from identifying the sender account.
NG-002  Preventing network, device, timing, behavioural or social-graph correlation.
NG-003  Reconstructing arbitrary photographs with perfect visual fidelity.
NG-004  Reliably detecting every proprietary watermark.
NG-005  Proving the absence of hidden information.
NG-006  Preserving every original style while claiming high assurance.
NG-007  Acting as a secure whistleblowing transport or anonymous network.
NG-008  Replacing legal, operational or organisational whistleblower protections.
NG-009  Uploading private content to a cloud AI service in the default build.
NG-010  Automatically changing ambiguous text without review.

2.3 Trust boundary

Trusted:
- the app’s signed code;
- bundled fonts and rendering assets;
- bundled normalization tables or version-pinned libraries;
- app-internal session storage;
- the pipeline engine;
- post-output verification code.

Untrusted:
- incoming Intent extras;
- MIME declarations;
- content-provider filenames;
- content-provider metadata;
- source image encoders;
- source text spans;
- HTML clipboard variants;
- OCR output;
- decoded QR or barcode values;
- URLs;
- user-selected optional plug-ins;
- third-party share targets;
- all source bytes and pixels.

Partially trusted:
- Android framework decoding;
- library parsers;
- platform font and shaping behaviour;
- system locale;
- device-specific image codecs.

Controls must minimize dependence on partially trusted components for high-assurance output.


====================================================================================================
3. THREAT MODEL
====================================================================================================
3.1 Adversary model

The adversary controls platform A and can provide different users with slightly different representations of apparently identical content. The adversary later obtains one or more shared copies from platform B and correlates them with candidate users.

The adversary may combine many weak observations rather than rely on a single visible marker.

Conceptual scoring model:

    Score(user) = sum over observations of evidence contributed by each retained feature.

The app’s primary defence is convergence:
Different source representations with the same intended visible meaning should produce the same, or substantially more similar, canonical output.

3.2 Threat channels

TH-FONT-001
Per-glyph font outline changes, kerning differences, hinting differences, anti-aliasing patterns or one-pixel character variations.

TH-SPATIAL-001
Small repeated blocks, rotation-synchronised patterns, tiling, alignment markers, local texture patterns or padding-dependent signals.

TH-FREQ-001
Signals embedded in frequency-domain coefficients or robust combinations of spatial and frequency domains.

TH-LSB-001
Least-significant-bit or low-amplitude colour-channel differences, possibly combined with robust higher-level patterns.

TH-ALPHA-001
Transparent or near-transparent alpha-channel content.

TH-UNICODE-001
Zero-width characters, default-ignorable characters, bidi controls, variation selectors, combining marks or unusual whitespace.

TH-CONFUSABLE-001
Latin, Greek, Cyrillic or other visually confusable characters; full-width and half-width variants; compatibility forms.

TH-PUNCT-001
Dash, quote, apostrophe, ellipsis, comma, colon, spacing and line-ending variants.

TH-META-001
EXIF, XMP, ICC, PNG chunks, WebP chunks, embedded thumbnails, encoder identifiers, timestamps, filenames and application fields.

TH-URL-001
Query parameters, fragments, path tokens, userinfo, personalised subdomains, redirect tokens, tracking parameters or high-entropy identifiers.

TH-LAYOUT-001
Different line wrapping, paragraph breaks, list ordering, indentation, card dimensions, bubble geometry or element spacing.

TH-SEMANTIC-001
Synonym choice, sentence order, punctuation style, intentionally altered spelling, platform-personalised content or generated paraphrases.

TH-BARCODE-001
QR codes or barcodes containing user-specific URLs, tokens or identifiers.

TH-REGION-001
Avatars, map tiles, notification icons, timestamps, account names, UI state, unique background content or embedded photographs.

TH-FILE-001
Container ordering, compression tables, chunk ordering, encoder fingerprints, colour profiles, quantisation patterns or file structure.

TH-CROSS-SAMPLE-001
Accumulation of multiple weak indicators across repeated captures or shares.

3.3 Threat-to-output expectations

Canonical Text:
- strongly severs source font, pixel, alpha, frequency and image-container channels;
- can normalize Unicode, punctuation and URL channels;
- may retain semantic personalization, wording, ordering and visible account data;
- depends on correct extraction and review.

Rebuilt Image:
- strongly severs source text glyph pixels when all text is redrawn;
- strongly severs source image container metadata;
- can sever source layout only when using canonical layout rather than exact geometry;
- may retain source pixels in user-approved regions;
- may retain semantic personalization.

Appearance-Preserving Image:
- may weaken simple metadata, alpha, LSB and pixel-grid signals;
- cannot claim reliable removal of robust watermarking;
- must be labelled lower assurance.

3.4 Security claim rule

No block is allowed to emit “safe”, “anonymous” or “watermark removed”.

Allowed language:
- source metadata was not copied;
- final output was re-scanned;
- text was serialized from a canonical representation;
- text regions were rendered with the bundled renderer;
- specified URL components were removed;
- specified unresolved findings remain;
- some output regions still depend on source pixels.


====================================================================================================
4. ASSURANCE MODEL
====================================================================================================
The assurance class is computed by the pipeline. The user cannot manually promote it.

AS-0  Unverified
Conditions:
- output has not completed all required verification blocks;
- a verification block failed;
- output was externally edited after generation;
- source dependency information is missing.

UI:
Red status.
Sharing may be allowed only after explicit acknowledgement.
No privacy claim is shown.

AS-1  Re-encoded Derivative
Conditions:
- final file was newly encoded;
- metadata was re-scanned;
- substantial source-pixel dependency remains.

UI:
“Source pixels retained.”
Suitable only as a convenience transformation.

AS-2  Reviewed Canonical Text
Conditions:
- final text came from the Canonical Document;
- Unicode and URL final scans passed;
- unresolved text ambiguities were reviewed or explicitly retained.

UI:
“Canonical text generated and verified.”

AS-3  Rebuilt Image with Source Regions
Conditions:
- text and UI primitives were freshly rendered;
- some image regions were retained from the source;
- every retained region is listed in the report;
- metadata and OCR round-trip checks completed.

UI:
“Image rebuilt; listed source regions retained.”

AS-4  Fully Rebuilt Textual Image
Conditions:
- all visible text came from canonical text;
- all UI primitives came from bundled renderer assets;
- no source image region is retained;
- output was reopened, metadata-scanned and OCR round-trip checked;
- unresolved findings are absent or explicitly accepted.

UI:
“Fully rebuilt from reviewed content.”

Assurance must be downgraded when:
- a source pixel region is added;
- an unknown optional block modifies output after verification;
- an external editor changes the artifact;
- final re-scan fails;
- OCR round-trip diverges from approved canonical text;
- a mandatory block is skipped due to failure.

6. DETAILED SCREEN SPECIFICATION
====================================================================================================
UX-SCR-001 — Entry

Purpose:
Start a session without asking for broad storage permission.

Layout:
- Product title and one-sentence scope.
- Primary button: Paste or enter text.
- Primary button: Choose image.
- Passive instruction: Content can also be shared to the app from another Android app.
- Privacy statement: Processing is local in the default build.
- Recent-session history is absent by default.
- Threat model link.

Behaviour:
- Incoming Android share bypasses this screen and enters import validation.
- If both text and image are supplied by a malformed Intent, show a source-choice gate rather than silently choosing.
- If multiple images are supplied, MVP rejects the session with a clear single-input explanation.
- Clear clipboard reference immediately after copying into session memory.

UX-SCR-002 — Text Input

Layout:
- Large plain-text editor.
- Code-point warning indicator.
- “Treat selected text as intentional language” action for multilingual content.
- Output selector in the header.
- Continue button.

Rules:
- Rich text is converted to plain text on import.
- Hidden characters are not silently removed at entry; they are highlighted in Review.
- Provide a reveal-control mode showing symbols for spaces, line endings and invisible characters.
- Do not auto-open external URLs.

UX-SCR-003 — Image Input Preview

Layout:
- Image preview.
- File format as detected from signature.
- Pixel dimensions as detected.
- Source metadata count.
- “Source is untrusted” label.
- Output selector.
- Continue button.

Rules:
- Do not display untrusted filename as the primary title.
- Do not load full-resolution bitmap merely to draw a thumbnail.
- Orientation preview must match the eventual decoded orientation.
- Animated formats are either rejected in MVP or flattened with an explicit warning.

UX-SCR-004 — Output Choice

Options:
A. Canonical Text
B. Rebuilt Image
C. Both
D. Appearance-Preserving Image — Experimental

Each option shows:
- what it preserves;
- what it discards;
- expected review burden;
- assurance ceiling.

The default for text input is Canonical Text.
The default for image input that is mostly text is Rebuilt Image.
The app may recommend an output but must not hide alternatives.

UX-SCR-005 — Preset Choice

Presets:
- Balanced
- Strict Text
- Strict URL
- Full Reconstruction
- Preserve Layout
- Experimental Derivative

A preset is a versioned ordered list of blocks and settings.
Preset cards show:
- intended content type;
- number of mandatory review categories;
- whether any source pixels can remain;
- whether URLs retain path/query;
- expected output type.

UX-SCR-006 — Sequential Workflow

Layout:
- Vertical list.
- A single execution line connects cards.
- No canvas panning.
- No arbitrary wires.
- Reorder handles only appear in Advanced mode.
- Header shows Input → Output.
- Footer shows Run, Pause/Cancel, Review and Export states.

Block card collapsed fields:
- Block ID
- Plain-language name
- status;
- findings count;
- changes count;
- warning count;
- mandatory or optional;
- input and output type icons.

Block card states:
WAITING
READY
RUNNING
REVIEW_REQUIRED
BLOCKED
DONE_NO_CHANGE
DONE_CHANGED
SKIPPED_BY_POLICY
FAILED_RECOVERABLE
FAILED_FATAL
INVALIDATED

Rules:
- A downstream setting change invalidates only affected later blocks.
- An upstream content change invalidates every dependent later block.
- Review-required halts execution before semantic output is finalized.
- Mandatory blocks cannot be deleted.
- Optional block disablement immediately updates assurance ceiling.

UX-SCR-007 — Block Detail

Tabs:
1. Purpose
2. Findings
3. Changes
4. Settings
5. Verification
6. Technical details

Purpose:
Describe the exact threat channel handled and the limits.

Findings:
Show original location, category, severity and confidence without implying guilt.

Changes:
Show before/after and a revert control where safe.

Settings:
Use constrained enums, not arbitrary scripts.

Verification:
Show what independent check will confirm the action.

Technical details:
Library version, block version, execution duration, input/output hashes stored only for the session, and errors. Never show sensitive raw data in logs.

UX-SCR-008 — Finding Review

Group findings by user decision:
- Meaning may change.
- Link may stop working.
- OCR is ambiguous.
- Source pixels would remain.
- Hidden formatting was found.
- Unknown data was found.
- No safe automatic rule exists.

Each review item supports:
- accept proposed change;
- retain source meaning;
- manually edit;
- mark as expected language;
- exclude region;
- view surrounding context.

Bulk actions are allowed only for homogeneous findings with the same rule.

UX-SCR-009 — Character Review

Display:
- visual glyph;
- Unicode name;
- code point;
- script;
- neighbouring characters;
- suggested canonical form;
- whether OCR engines disagreed;
- confusable skeleton result;
- impact on URLs or identifiers.

Never replace confusable characters globally without language policy.

UX-SCR-010 — URL Review

Display:
- original display text;
- parsed scheme;
- host;
- registrable domain;
- subdomain;
- path;
- query;
- fragment;
- userinfo;
- suspected high-entropy components;
- cleaned result;
- predicted functionality impact.

Actions:
- keep full URL;
- remove known tracking fields;
- remove all query and fragment;
- keep origin only;
- convert to non-clickable domain text;
- remove URL entirely;
- manually edit.

UX-SCR-011 — Layout Review

Display:
- reading order numbers;
- OCR block boxes;
- line groups;
- proposed canonical layout;
- source layout overlay.

Actions:
- reorder blocks;
- merge;
- split;
- mark decorative;
- mark hidden;
- lock exact wording;
- choose generic card, message bubble, article or plain document renderer.

UX-SCR-012 — Image Region Review

Every non-text region receives one policy:
- Replace with generic placeholder.
- Solid redact.
- Crop.
- Rebuild from structured data.
- Retain source pixels.
- Remove.

Retaining source pixels:
- requires explicit action;
- lowers assurance;
- shows a persistent output overlay until acknowledged;
- appears in the final report.

UX-SCR-013 — Output Preview

Tabs:
- Canonical Text
- Rebuilt Image
- Derivative Image, when enabled
- Diff
- Verification

The Share button is disabled until mandatory verification completes, unless the user chooses an unverified emergency export that is clearly marked.

UX-SCR-014 — Verification Report

Summary:
- final assurance class;
- executed block versions;
- findings detected;
- changes applied;
- unresolved findings;
- retained source regions;
- final metadata inventory;
- final Unicode scan;
- final URL scan;
- OCR round-trip differences;
- whether the output was rebuilt or derived.

The report itself must not contain source secrets beyond what is necessary for the user to understand decisions.

UX-SCR-015 — Android Share

Use the system Sharesheet.
Do not implement a custom destination list.
Share:
- text/plain for canonical text;
- a content URI with temporary read permission for images;
- both only when the selected target can accept the intended combination.

UX-SCR-016 — Explicit Save

Saving is a separate user decision from sharing.
Default:
- no permanent source copy;
- no permanent session history;
- no permanent report.

Saved output:
- uses a user-visible filename generated by the app;
- never reuses the untrusted source filename;
- optionally saves a separate report, never embeds the full report in image metadata.

UX-SCR-017 — Session Cleanup

Show:
- source cache deletion;
- OCR derivative deletion;
- render cache deletion;
- output cache deletion after share permission expiry policy;
- report memory release.

The UI should say “Session cleared” only after deletion attempts complete.
Failures are listed without exposing content.

UX-SCR-018 — Workflow Preset Editor

Sequence-only editor:
- ordered cards;
- add optional block from a filtered palette;
- move only to type-compatible positions;
- validation runs after every edit;
- mandatory blocks are pinned;
- assurance ceiling updates live;
- preset schema version displayed.

No arbitrary custom code block in production.
No user-provided executable scripts.
No network block in the default edition.

UX-SCR-019 — Threat Coverage

Matrix:
- threat channel;
- blocks addressing it;
- output modes affected;
- known limitations;
- last verified test corpus version.

UX-SCR-020 — Error Recovery

Recoverable examples:
- OCR model unavailable;
- memory budget exceeded;
- a URI grant expired;
- an optional scan failed.

Fatal examples:
- source cannot be safely decoded;
- Canonical Document schema invalid;
- output re-open verification fails;
- mandatory block was bypassed;
- a final artifact references the source URI.

Recovery UI:
- retry block;
- choose lower-cost preset;
- return to review;
- discard session.
Do not silently continue after a mandatory fatal error.

7. SEQUENCE-ONLY PIPELINE ENGINE
====================================================================================================
7.1 Execution Context

Every block receives one immutable ExecutionContext and returns a new ExecutionContext.

Conceptual fields:

ExecutionContext
- sessionId: ephemeral random value
- workflowId
- workflowVersion
- executionRevision
- inputKind: TEXT | IMAGE
- requestedOutput: TEXT | REBUILT_IMAGE | BOTH | DERIVATIVE_IMAGE
- sourceHandle: restricted internal reference
- canonicalDocument: optional
- textArtifact: optional
- imageArtifact: optional
- derivativeArtifact: optional
- findings: append-only logical collection
- decisions: user-approved decision collection
- changes: append-only ledger
- warnings: append-only collection
- sourceDependencyMap
- verificationReport
- assuranceCeiling
- currentAssurance
- lifecycleState

No block receives arbitrary access to another block’s private files.
Blocks request artifacts through typed handles.

7.2 Block contract

Required metadata:
- blockId
- blockVersion
- displayName
- description
- inputPredicate
- outputGuarantee
- mandatory
- deterministic
- requiresReview
- settingsSchemaVersion
- threatCoverage
- invalidationKeys
- verificationRequirements
- resourceClass
- offlineCapability
- licenceMetadata for third-party code

Required methods:
- validateConfiguration
- inspect
- planChanges
- apply
- summarize
- cleanup
- produceVerificationHints

Block phases:
1. Validate
2. Inspect
3. Plan
4. Review when required
5. Apply
6. Self-check
7. Commit new context
8. Cleanup temporary block artifacts

7.3 Pipeline validation rules

PV-001  Exactly one source block must exist.
PV-002  Exactly one final export preparation block must exist.
PV-003  No cycle is representable.
PV-004  Every block input predicate must be satisfied by all previous committed state.
PV-005  Mandatory blocks cannot be removed.
PV-006  Verification must occur after final serialization.
PV-007  Any transformation after verification invalidates verification.
PV-008  A derivative-image block sets an assurance ceiling.
PV-009  Retaining source pixels sets an assurance ceiling.
PV-010  A URL serializer must occur after URL policy.
PV-011  Canonical text serialization must occur after final character review.
PV-012  Image rendering must occur after Canonical Document approval.
PV-013  Metadata re-scan must inspect the actual saved output bytes.
PV-014  OCR round-trip must inspect the final image artifact, not the render canvas.
PV-015  Export cannot claim a higher assurance than the computed ceiling.
PV-016  A failed mandatory block blocks verified export.
PV-017  A workflow preset must declare its compatible input and output types.
PV-018  Unknown block versions invalidate preset signatures until migrated.
PV-019  External plug-ins cannot run in the high-assurance profile.
PV-020  A block cannot write persistent logs containing source content.

7.4 Invalidation rules

Change to source:
Invalidate every downstream block.

Change to language policy:
Invalidate Unicode analysis, canonical text, URL display decisions, rendering and verification.

Change to URL policy:
Invalidate URL changes, text serialization, image rendering and verification.

Change to one review decision:
Invalidate the Canonical Document revision and every artifact derived from that revision.

Change to renderer settings:
Invalidate image artifacts and image verification, but not approved canonical text.

Change to save filename:
Does not invalidate content verification, but filename-safety validation must rerun.

7.5 Cancellation

Cancellation must:
- stop new block execution;
- allow the currently executing native decoder to return safely;
- mark partial artifacts invalid;
- release bitmaps and model handles;
- delete block-local temporary files;
- leave the session in a resumable or discardable state;
- never export partially verified output as verified.

9. DETAILED BLOCK CATALOG
====================================================================================================
The following catalog is normative for IDs and responsibilities.
A block ID must never be reused for a materially different action.
A future incompatible behaviour requires a new block version or a new ID.



[SYS-001] Create Ephemeral Session
Stage:
Session

Input:
Android Intent, Photo Picker result, clipboard paste or editor submission

Output:

11.2 Add block behaviour

====================================================================================================
12. VERIFICATION STRATEGY
====================================================================================================
12.1 Verification principles

V-P-001  Inspect final artifacts, not merely function return values.
V-P-002  Use an independent parser or code path where practical.
V-P-003  Treat OCR as untrusted and verify rendering with round-trip comparison.
V-P-004  Treat absence of detector findings as “not detected”, not “absent”.
V-P-005  Every claim shown in the UI maps to a verification result.
V-P-006  Every semantic change maps to a change-ledger entry.
V-P-007  Every retained source region maps to a user decision.
V-P-008  Every workflow has a tested assurance ceiling.
V-P-009  Verification failure downgrades or blocks; it never silently retries with weaker rules.
V-P-010  Release tests include malicious and malformed inputs.

12.2 Required verification layers

Layer A — Structural
- workflow manifest valid;
- required blocks present;
- block versions recognized;
- output revision current.

Layer B — Content
- final Unicode scan;
- final URL parse;
- OCR round-trip;
- barcode re-scan;
- reading-order comparison.

Layer C — Container
- actual metadata inventory;
- embedded thumbnail absence;
- channel/alpha model;
- allowed chunk set;
- output decodes independently.

Layer D — Provenance
- no source URI;
- no untrusted filename;
- no source provider authority;
- no unexplained source pixels;
- no stable session/device seed.

Layer E — Privacy engineering
- offline execution;
- sensitive logging audit;
- app-private storage;
- cleanup success;
- temporary share permission scope.

12.3 Verification status vocabulary

PASS
The specified check completed and matched its approved condition.

PASS_WITH_DECLARED_RESIDUAL
The check completed and a documented residual is intentionally retained.

REVIEW_REQUIRED
The check found an ambiguity requiring user input.

NOT_APPLICABLE
The check does not apply to this output.

NOT_RUN
The check did not execute. This prevents a verified assurance class when required.

FAIL
The specified condition was violated.

ERROR
The verifier could not establish a result. Treat as failure for assurance purposes.

12.4 No-detector rule

Frequency, LSB or watermark detectors may be useful comparative diagnostics, but:
- they do not certify cleanliness;
- their thresholds require corpus calibration;
- they are not mandatory for canonical text or fully rebuilt textual images;
- they cannot promote assurance;
- a detector miss is not evidence of absence.

14. QUALITY AND ACCEPTANCE CRITERIA
====================================================================================================
The project must not use arbitrary quality thresholds chosen by a coding agent.
Thresholds for OCR consensus, diagnostic detectors, image quality and resource budgets must be derived from the maintained test corpus and documented benchmark runs.

Functional acceptance:

AC-F-001  Text share enters as TEXT without requiring media permission.
AC-F-002  Image share enters as IMAGE through a content URI.
AC-F-003  The selected output type is always visible.
AC-F-004  A workflow cannot form a branch or cycle.
AC-F-005  Mandatory blocks cannot be removed.
AC-F-006  Invalid block order is rejected before execution.
AC-F-007  Every semantic change appears in the diff.
AC-F-008  Every retained source region appears in the report.
AC-F-009  Text and image outputs in BOTH mode share one canonical revision.
AC-F-010  Share uses the system Sharesheet.
AC-F-011  Save is explicit and separate from share.
AC-F-012  Session purge removes transient artifacts.

Security acceptance:

AC-S-001  Strict workflows complete without network.
AC-S-002  No raw source data appears in normal logs.
AC-S-003  No output exposes the source URI or filename.
AC-S-004  Output image metadata matches the allowlist after re-scan.
AC-S-005  Final text contains no unapproved invisible controls.
AC-S-006  Final URLs match approved parsed components.
AC-S-007  Rebuilt-image source dependencies are complete and accurate.
AC-S-008  Verification failure cannot show a high assurance badge.
AC-S-009  Derivative mode cannot exceed derivative assurance.
AC-S-010  External plug-ins cannot enter the high-assurance preset.
AC-S-011  Bundled renderer never silently falls back to a device font.
AC-S-012  Random perturbation never uses a stable device-derived seed.

UX acceptance:

AC-U-001  A non-expert can identify input type, output type and current stage.
AC-U-002  “Detected”, “Changed” and “Verified” are visually distinct.
AC-U-003  URL changes show functionality warnings.
AC-U-004  Character review exposes code point and script details in expert disclosure.
AC-U-005  Source-pixel retention is visible before export.
AC-U-006  No screen claims guaranteed anonymity.
AC-U-007  Every fatal error offers discard and a safe alternative output when feasible.
AC-U-008  TalkBack/accessible labels describe block status and review actions.
AC-U-009  Colour is not the only status indicator.
AC-U-010  Large text and landscape layouts remain usable.

Agent acceptance:

AC-A-001  Each pull request changes one bounded work package.
AC-A-002  New dependencies include licence, maintenance and offline analysis.
AC-A-003  New blocks include contract, tests, invalidation rules and verification.
AC-A-004  Canonical schema changes include migration tests.
AC-A-005  Security-sensitive behaviour includes adversarial tests.
AC-A-006  No agent implements a custom Unicode confusable database, URL parser, OCR engine or image codec without explicit approval.

19. PROHIBITED IMPLEMENTATION PATTERNS
====================================================================================================
ANTI-001  One “AI clean everything” block.
Reason:
Cannot explain, verify or isolate failure.

ANTI-002  Gaussian noise as the primary defence.
Reason:
Does not guarantee removal of robust signals.

ANTI-003  Blur or pixelation as secure deletion.
Reason:
May retain recoverable information and source structure.

ANTI-004  Copying source bitmap then drawing over it for “reconstruction”.
Reason:
Hidden source pixels may remain and dependency is unclear.

ANTI-005  Preserving source metadata then deleting only GPS.
Reason:
Other fields and chunks may carry identifiers.

ANTI-006  Regex-only URL parsing.
Reason:
Cannot safely model all URL components and encodings.

ANTI-007  Global Greek/Cyrillic-to-Latin replacement.
Reason:
Corrupts legitimate language and still misses contextual attacks.

ANTI-008  NFKC-only Unicode defence.
Reason:
Does not solve all confusables, controls, punctuation and semantic differences.

ANTI-009  OCR result accepted without review or consensus.
Reason:
OCR can normalize correctly, normalize incorrectly or retain a confusable.

ANTI-010  Device-font rendering in high-assurance output.
Reason:
May create device-specific output and change glyph behaviour.

ANTI-011  Stable per-installation random noise.
Reason:
Creates a new repeatable fingerprint.

ANTI-012  Cloud OCR silently used as fallback.
Reason:
Changes the trust boundary and may leak content.

ANTI-013  Trusting incoming filename or MIME.
Reason:
Both are attacker-controlled.

ANTI-014  Verifying configuration rather than final bytes.
Reason:
Encoder or serializer behaviour may differ from intent.

ANTI-015  Saving source by default.
Reason:
Creates unnecessary sensitive persistence.

ANTI-016  Custom Sharesheet.
Reason:
Recreates platform behaviour and may introduce destination tracking.

ANTI-017  Assurance chosen by the user.
Reason:
Assurance is evidence-based, not a preference.

ANTI-018  Detector miss shown as “clean”.
Reason:
No detector covers all unknown channels.

ANTI-019  Agent-generated thresholds without a benchmark.
Reason:
Creates unsupported numeric security claims.

ANTI-020  External plug-in in high-assurance workflow.
Reason:
Breaks control over code, data access and verification.

22. FINAL PRODUCT RECOMMENDATION
====================================================================================================
The sequence-only ComfyUI-like concept is approved with the following constraint:

The user sees an ordered, inspectable workflow, but the app remains a typed security pipeline rather than a general visual-programming environment.

The essential product loop is:

INPUT
  Text or Image
    ↓
INSPECT
  Unicode, URL, metadata, OCR, layout and regions
    ↓
CANONICALIZE
  Build reviewed semantic representation
    ↓
RENDER
  Canonical text and/or fresh rebuilt image
    ↓
VERIFY
  Reopen final artifacts and compute assurance
    ↓
SHARE
  Android system Sharesheet
    ↓
PURGE
  Delete transient session data

The strongest design decision is not adding more filters.
It is ensuring that outputs are generated from an approved Canonical Document rather than inherited source representation.

The app should continuously communicate three separate facts:

1. What was detected.
2. What was changed.
3. What was verified.

No interface element should collapse those three states into a single “clean” button.
